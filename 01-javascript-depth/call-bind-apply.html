手写 call / apply / bind

JavaScript 深度 · Day 9 · 碎片时间+晚上动手写

为什么要学这个

在 JavaScript 中，this 的指向取决于函数的调用方式，而不是定义位置。这导致一个经典问题： 

const timer = {
  seconds: 0,
  tick() { this.seconds++ }
}

setInterval(timer.tick, 1000)
// this.seconds++ 报错！
// 因为 setInterval 里 this 指向 window，不是 timer

call / apply / bind 就是 JS 提供的三把工具，让你手动指定 this 到底指向谁。

三者对比

方法 执行时机 参数格式 返回值
call 立即执行 fn.call(ctx, a, b, c) 函数的返回值
apply 立即执行 fn.apply(ctx, [a, b, c]) 函数的返回值
bind 不执行 fn.bind(ctx, a)(b, c) 一个新函数
记忆口诀：
call → 打电话，马上接，一个一个说
apply → 申请表，马上交，信息打包一份交
bind → 绑定，先绑好，以后再打

核心原理

this 到底怎么决定的？

JS 里 this 的指向遵循一条铁律：「谁调用函数，this 就指向谁」 

obj.method()   // this === obj     （obj 在调用）
method()       // this === window  （没人调用 = window 调用）
new Foo()      // this === 新实例    （new 创建的对象）
fn.call(obj)   // this === obj     （手动指定）

call 的底层思路

既然 obj.method() 的 this 会指向 obj，那只要：

把 fn 挂到 obj 上 → obj.fn() 调用 → this === obj → delete obj.fn 
// 本质上，greet.call(obj, 'Hi', '!') 等价于：
obj.__temp__ = greet             // Step 1: 挂上去
const result = obj.__temp__('Hi', '!')  // Step 2: 作为方法调用 → this === obj
delete obj.__temp__              // Step 3: 清理
问题： 如果 obj 上已经有一个属性叫 __temp__，就会被覆盖！
解法： 用 Symbol() 做 key —— 每次生成的 Symbol 都是全局唯一的，永远不会和已有属性冲突。

手写实现

Step 1 · 手写 call 
Function.prototype.myCall = function (context, ...args) {
  // 1. context 为 null/undefined → 指向全局对象
  context = context ?? globalThis

  // 2. 原始值（数字/字符串/布尔）需要包装成对象
  //    因为原始值上不能挂属性
  if (typeof context !== 'object' && typeof context !== 'function') {
    context = Object(context)
  }

  // 3. 创建唯一 key
  const key = Symbol('tempFn')

  // 4. this 就是调用 myCall 的那个函数
  //    greet.myCall(obj) → this === greet
  context[key] = this

  // 5. 通过 context.method() 调用 → this 指向 context
  const result = context[key](...args)

  // 6. 清理临时属性
  delete context[key]

  return result
}
为什么用 Symbol 而不是普通字符串作 key？
Symbol 保证全局唯一。如果用 '__tempFn__'，万一 context 上已有同名属性，就会被覆盖 → 产生 bug 并且很难排查。Symbol 是零冲突方案。
为什么 this 就是 greet 函数？
还是那条规则：「谁调用，this 就指向谁」。greet.myCall(obj) 里，myCall 是通过 greet 调用的 → 所以 myCall 内部的 this 就是 greet。
为什么要处理原始值（Object(context)）？
原始值（如数字 42）不是对象，不能在上面挂属性。42[key] = fn 会静默失败。Object(42) 会把它包装成 Number {42} 对象，就可以挂属性了。原生 call 就是这么处理的。
Step 2 · 手写 apply
和 call 只有一个区别：第二个参数是数组，不是展开的散列参数。
Function.prototype.myApply = function (context, argsArray) {
  context = context ?? globalThis
  if (typeof context !== 'object' && typeof context !== 'function') {
    context = Object(context)
  }

  const key = Symbol('tempFn')
  context[key] = this

  // 唯一区别：argsArray 可能不传，需要兜底
  const result = argsArray
    ? context[key](...argsArray)
    : context[key]()

  delete context[key]
  return result
}
为什么 ES6 之后 apply 用得少了？
因为展开运算符 ... 直接替代了它唯一的优势： 
// ES5: 展开数组只能靠 apply
Math.max.apply(null, [3, 1, 4, 1, 5])   // 5

// ES6+: 直接展开
Math.max(...[3, 1, 4, 1, 5])           // 5
重点 Step 3 · 手写 bind

bind 和 call/apply 的本质区别

call / apply → 立即执行，用完就完
bind → 不执行，返回新函数 → 新函数的 this 被永久锁定 
const obj = { name: 'Bob' }

function greet(greeting, punct) {
  return greeting + ', ' + this.name + punct
}

// call: 立即出结果
greet.call(obj, 'Hi', '!')     // "Hi, Bob!"

// bind: 先拿到新函数，以后想用几次用几次
const boundGreet = greet.bind(obj)
boundGreet('Hi', '!')          // "Hi, Bob!"
boundGreet('Hey', '~')         // "Hey, Bob~"

偏函数（Partial Application）

bind 可以在绑定 this 的同时预填一部分参数，剩下的在调用时再传： 

const sayHey = greet.bind(obj, 'Hey')  // 预填 greeting = 'Hey'
sayHey('!')    // "Hey, Bob!"   ← 只需传 punct
sayHey('~')    // "Hey, Bob~"

// 实际上发生了什么：
//   bind 时传了 ['Hey']        → outerArgs
//   调用时传了 ['!']           → innerArgs
//   最终拼成    ['Hey', '!']   → 传给原函数

基础版实现

Function.prototype.myBind = function (context, ...outerArgs) {
  // this = 原函数（greet.myBind → this === greet）
  const originalFn = this

  // 返回一个新函数
  return function boundFn(...innerArgs) {
    // 把两次传入的参数拼起来，用 call 指定 this
    return originalFn.call(context, ...outerArgs, ...innerArgs)
  }
}
为什么 const originalFn = this 要在外面保存？
因为返回的 boundFn 被调用时，它内部的 this 已经不是原函数了（可能是 window 或调用者）。所以必须在外层闭包里把原函数存下来。这就是经典的闭包保存外层变量模式。

进阶 支持 new 的完整版

原生 bind 返回的函数可以被 new 调用。此时 this 应该指向新实例，而不是 bind 时的 context： 

Function.prototype.myBind = function (context, ...outerArgs) {
  const originalFn = this

  function boundFn(...innerArgs) {
    // 关键判断：是 new 调用还是普通调用
    const isNew = this instanceof boundFn
    return originalFn.call(
      isNew ? this : context,   // new → 用新实例；普通 → 用 context
      ...outerArgs,
      ...innerArgs
    )
  }

  // 继承原函数的原型链，让 instanceof 正常工作
  boundFn.prototype = Object.create(originalFn.prototype)
  return boundFn
}
面试只需要写到基础版还是完整版？
先写基础版，写完后主动说「如果要支持 new 调用还需要判断 this instanceof boundFn」——不用真写出来，但提到这个知识点就是加分项。面试官看的是你知不知道这个边界，不是能不能默写。

实战场景速查

场景 1 · 判断精确类型（call）

typeof 只能区分原始类型，对象一律返回 'object'。要精确判断得借用 Object.prototype.toString：

Object.prototype.toString.call([])        // "[object Array]"
Object.prototype.toString.call(null)      // "[object Null]"
Object.prototype.toString.call(undefined) // "[object Undefined]"
Object.prototype.toString.call(new Date)  // "[object Date]"
Object.prototype.toString.call(/re/)     // "[object RegExp]"
场景 2 · 类数组转真数组（call）

arguments、NodeList 等类数组没有数组方法，借用 Array 的：

// ES5 经典写法
function foo() {
  var args = Array.prototype.slice.call(arguments)
  // arguments 没有 .slice()，借 Array 的来用
}

// ES6+ 直接用：
const args = [...arguments]
// 或
const args = Array.from(arguments)
场景 3 · 事件回调锁定 this（bind）
class Timer {
  constructor() {
    this.seconds = 0

    // 方法 A: bind
    setInterval(this.tick.bind(this), 1000)

    // 方法 B: 箭头函数（ES6+ 更常用）
    setInterval(() => this.tick(), 1000)
  }
  tick() { this.seconds++ }
}
场景 4 · 预填参数 / 偏函数（bind）
function log(level, module, msg) {
  console.log(`[${level}] ${module}: ${msg}`)
}

// 预填 level 和 module，生成专用 logger
const dbWarn  = log.bind(null, 'WARN', 'DB')
const apiInfo = log.bind(null, 'INFO', 'API')

dbWarn('connection timeout')   // "[WARN] DB: connection timeout"
apiInfo('request received')   // "[INFO] API: request received"
场景 5 · React 类组件（bind）

React 类组件中事件处理函数的 this 不会自动绑定：

class App extends React.Component {
  constructor(props) {
    super(props)
    // 不 bind 的话 onClick 触发时 this 是 undefined
    this.handleClick = this.handleClick.bind(this)
  }
  handleClick() { console.log(this.state) }
}

// 现代写法：箭头函数 class field 自动绑定
class App extends React.Component {
  handleClick = () => { console.log(this.state) }
}

面试话术

当面试官问「手写 call / apply / bind」

「call 和 apply 的核心思路一样：把目标函数临时挂到 context 上作为方法调用，这样 this 就自然指向 context 了。调完用 delete 清掉。用 Symbol 做 key 避免属性冲突。两者唯一区别就是参数格式——call 逐个传，apply 传数组。」
「bind 不同于前两者，它返回一个新函数而不是立即执行。核心是用闭包保存 context 和预传参数，调用时再把两批参数拼起来走 call。如果要完整实现还需要处理 new 的情况——new 的优先级高于 bind，this 应该指向新实例。」

加分回答清单

  • 提到 Symbol 防属性冲突（而不是用 '__temp__'）
  • 提到 context ?? globalThis 而不是 context || window（严格模式兼容）
  • 提到原始值需要 Object() 包装
  • 提到 bind 是偏函数（partial application），和柯里化的关系
  • 提到 bind + new 的优先级：this instanceof boundFn
  • 提到 ES6 箭头函数已经能覆盖大部分 bind 的场景

常见追问

call 和 apply 性能有区别吗？
现代引擎差异可以忽略。V8 内部对两者都做了优化。选择哪个取决于你的参数是散列的还是数组——有数组用 apply（或直接展开），没数组用 call。
箭头函数能用 call/apply/bind 改 this 吗？
不能。箭头函数没有自己的 this，它的 this 在定义时就从外层词法作用域继承了，无法被 call/apply/bind 覆盖。这也是箭头函数能"自动绑定"的原因。
bind 两次会怎样？fn.bind(a).bind(b)()
this 指向 a，不是 b。第一次 bind 返回的新函数内部已经用闭包锁死了 context = a。第二次 bind 虽然又包了一层，但调用时走到最里面还是 originalFn.call(a, ...)。所以 bind 只有第一次生效。
完整测试代码 
const obj = { name: '张三' }

function greet(greeting, punctuation) {
  return greeting + ', ' + this.name + punctuation
}

// ── 测试 myCall ──
console.log(greet.myCall(obj, 'Hello', '!'))
// => "Hello, 张三!"

// ── 测试 myApply ──
console.log(greet.myApply(obj, ['Hi', '.']))
// => "Hi, 张三."

// ── 测试 myBind ──
const bound = greet.myBind(obj, 'Hey')
console.log(bound('~'))
// => "Hey, 张三~"

// ── 测试 myBind: 不预填参数 ──
const bound2 = greet.myBind(obj)
console.log(bound2('Yo', '!'))
// => "Yo, 张三!"

// ── 测试 context = null ──
function getGlobal() { return this }
console.log(getGlobal.myCall(null) === globalThis)
// => true
