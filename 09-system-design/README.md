## 🎯 核心方法论：4 步框架

每道系统设计题都按这个框架答，**形成肌肉记忆**：

```
1. 需求澄清（30s）
   - 用户量级？数据量级？性能要求？
   - 移动端 / 桌面端 / 跨端？
   
2. 整体架构（2min）
   - 数据流
   - 模块划分（前端 / API / 数据库 / 缓存）
   - 关键技术选型
   
3. 关键问题深挖（5min）
   - 性能瓶颈 + 优化方案
   - 边界 case + 容错
   - 扩展性
   
4. 收尾权衡（30s）
   - 这个方案的缺点
   - 不同场景下的备选方案
```

## 🎯 必备 5 题（背框架，不背答案）

### 题 1：无限滚动列表（虚拟滚动）

```
核心问题：1 万条数据如何不卡？
关键技术：
- 可视区域渲染（计算 startIndex/endIndex）
- absolute 定位 + transform translate
- 懒加载图片（IntersectionObserver）
- 高度估算（动态 vs 固定）
难点：动态高度怎么处理？
答案：缓存已渲染项的实际高度，未渲染用估算
```

### 题 2：商详页性能优化（你的强项）

```
关键路径：
- 首屏关键 CSS 内联
- 图片：WebP/AVIF + 多尺寸 + LQIP（低质占位）
- 数据：预渲染（SSR/SSG/ISR） + 接口聚合
- 资源：路由级 code splitting + preload
量化数据：LCP 从 X 降到 Y，转化率提升 Z%
```

### 题 3：购物车系统

```
核心问题：
- 已登录 vs 未登录（localStorage vs DB）
- 多端同步（登录后合并）
- 限购 / 库存校验
- 价格计算（优惠/满减/会员）
难点：价格计算在前端做还是后端做？
答案：后端做（前端可篡改），前端只展示
```

### 题 4：AI Chatbot 完整架构（自己项目，最详细）

```
前端：
- React Streaming（SSE）
- 历史对话本地缓存（IndexedDB）
- Markdown + 代码高亮渲染
后端：
- Next.js API Route / Server Action
- 调 LLM API（带重试 + 限流）
- RAG：向量库检索 → 拼 prompt
- Tool Use：函数路由 + 执行 + 回写
存储：
- 对话历史 → Postgres
- 文档向量 → pgvector
难点：
- Token 限制：滑窗策略（保留最近 N 轮 + 用 LLM 总结历史）
- 成本控制：缓存重复 query / 用便宜模型预筛
- 流式中断：用户取消时怎么释放后端连接
```

### 题 5：错误监控系统（前端基建）

```
核心：捕获 + 聚合 + 上报 + 告警
捕获：
- window.onerror / unhandledrejection
- React ErrorBoundary
- 资源加载错误（addEventListener('error', true)）
聚合：
- 错误指纹（message + stack 哈希）
- 同样错误合并，记录次数
上报：
- 防抖（不要每错都发）
- sendBeacon（页面关闭也能发）
- 失败重试 + 队列
告警：
- 同错误 1 分钟超 X 次 → 报警
- 错误率突增 → 报警
```

## 🎤 面试话术

### Q: 设计一个无限滚动列表
> 先澄清：数据量？item 高度固定吗？
> 
> **核心方案**：虚拟滚动。可视区高度 H、item 高度 h，可见数 N=H/h，加缓冲区共渲染 N+10 个。滚动时算 startIndex 决定渲染哪一段，外层用空 div 撑总高度，内层 absolute 定位到对应位置。
> 
> 关键优化：
> 1. transform 而非 top 定位（不触发回流）
> 2. 图片用 IntersectionObserver 懒加载
> 3. 动态高度：缓存已测量项，未测量用估算
> 
> 我之前在电商商品列表里做过这个方案，从 5fps 提升到 60fps。

### Q: 你的 AI Chatbot 架构？
> 看图说话（出图）。
> 前端用 Next.js + AI SDK，SSE 流式渲染。
> 后端 API Route 调 Claude，加了 RAG（Supabase pgvector）和 1 个 Tool Use（X 功能）。
> 重点讲 3 个难点：
> 1. **Token 限制**：滑窗 + 历史总结
> 2. **成本控制**：相似 query 缓存 / 用 Haiku 预筛
> 3. **流式中断**：客户端 abort signal → 后端 stream cancel

### Q: 怎么做前端错误监控？
> 三步：捕获 → 聚合 → 上报。
> 捕获要全：JS 错误（onerror）、Promise 错误（unhandledrejection）、React 错误（ErrorBoundary）、资源错误（事件捕获 phase）。
> 聚合靠错误指纹（message+stack 哈希），同错误合并计数。
> 上报用 sendBeacon（不阻塞页面），失败入本地队列下次重传。
> 我们生产用的是 Sentry，但原理是这个。
